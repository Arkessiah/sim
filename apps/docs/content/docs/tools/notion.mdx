[Notion](https://www.notion.so) is an all-in-one workspace that combines notes, documents, wikis, and project management tools into a single platform. It offers a flexible and customizable environment where users can create, organize, and collaborate on content in various formats.

With Notion, you can:

- **Create versatile content**: Build documents, wikis, databases, kanban boards, calendars, and more
- **Organize information**: Structure content hierarchically with nested pages and powerful databases
- **Collaborate seamlessly**: Share workspaces and pages with team members for real-time collaboration
- **Customize your workspace**: Design your ideal workflow with flexible templates and building blocks
- **Connect information**: Link between pages and databases to create a knowledge network
- **Access anywhere**: Use Notion across web, desktop, and mobile platforms with automatic syncing

In Sim, the Notion integration enables your agents to interact directly with your Notion workspace programmatically. This allows for powerful automation scenarios such as knowledge management, content creation, and information retrieval. Your agents can:

- **Read Notion pages**: Extract content and metadata from any Notion page.
- **Read Notion databases**: Retrieve database structure and information.
- **Write to pages**: Append new content to existing Notion pages.
- **Create new pages**: Generate new Notion pages under a parent page, with custom titles and content.
- **Query databases**: Search and filter database entries using advanced filter and sort criteria.
- **Search workspace**: Search across your entire Notion workspace for pages or databases matching specific queries.
- **Create new databases**: Programmatically create new databases with custom properties and structure.

This integration bridges the gap between your AI workflows and your knowledge base, enabling seamless documentation and information management. By connecting Sim with Notion, you can automate documentation processes, maintain up-to-date information repositories, generate reports, and organize information intelligentlyâ€”all through your intelligent agents.

## Operations

### `notion_read`

Read content from a Notion page

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `pageId` | string | Yes | The ID of the Notion page to read |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Page content in markdown format with headers, paragraphs, lists, and todos |
| `metadata` | object | Page metadata including title, URL, and timestamps |

### `notion_read_database`

Read database information and structure from Notion

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `databaseId` | string | Yes | The ID of the Notion database to read |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Database information including title, properties schema, and metadata |
| `metadata` | object | Database metadata including title, ID, URL, timestamps, and properties schema |

### `notion_write`

Append content to a Notion page

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `pageId` | string | Yes | The ID of the Notion page to append content to |
| `content` | string | Yes | The content to append to the page |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Success message confirming content was appended to page |

### `notion_create_page`

Create a new page in Notion

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `parentId` | string | Yes | ID of the parent page |
| `title` | string | No | Title of the new page |
| `content` | string | No | Optional content to add to the page upon creation |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Success message confirming page creation |
| `metadata` | object | Page metadata including title, page ID, URL, and timestamps |

### `notion_query_database`

Query and filter Notion database entries with advanced filtering

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `databaseId` | string | Yes | The ID of the database to query |
| `filter` | string | No | Filter conditions as JSON \(optional\) |
| `sorts` | string | No | Sort criteria as JSON array \(optional\) |
| `pageSize` | number | No | Number of results to return \(default: 100, max: 100\) |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Formatted list of database entries with their properties |
| `metadata` | object | Query metadata including total results count, pagination info, and raw results array |

### `notion_search`

Search across all pages and databases in Notion workspace

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `query` | string | No | Search terms \(leave empty to get all pages\) |
| `filterType` | string | No | Filter by object type: page, database, or leave empty for all |
| `pageSize` | number | No | Number of results to return \(default: 100, max: 100\) |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Formatted list of search results including pages and databases |
| `metadata` | object | Search metadata including total results count, pagination info, and raw results array |

### `notion_create_database`

Create a new database in Notion with custom properties

#### Input

| Parameter | Type | Required | Description |
| --------- | ---- | -------- | ----------- |
| `parentId` | string | Yes | ID of the parent page where the database will be created |
| `title` | string | Yes | Title for the new database |
| `properties` | string | No | Database properties as JSON object \(optional, will create a default "Name" property if empty\) |

#### Output

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `content` | string | Success message with database details and properties list |
| `metadata` | object | Database metadata including ID, title, URL, creation time, and properties schema |



## Best Practices

{/* MANUAL-CONTENT-START:bestPractices */}
When incorporating the Notion block into your Sim.ai workflows, following established best practices ensures reliable content management and optimal performance. The Notion block serves as a comprehensive content management bridge between your automated workflows and Notion workspaces, enabling seamless document creation, content retrieval, and database operations that transform static information into dynamic, workflow-driven knowledge systems.

**Connection Tag Usage and Data Flow**

Effective connection tag implementation forms the backbone of successful Notion integrations. The `<notion1.content>` output provides the primary data stream for retrieved content, formatted search results, or operation confirmations. When chaining Notion operations, utilize `<notion1.metadata.pageId>` from creation operations as input for subsequent read or write operations, creating seamless content lifecycle workflows.

For dynamic page creation, leverage AI-generated content by connecting `<agent1.content>` to `<notion2.content>` input parameters. This enables automated documentation generation where AI processes information and directly creates or updates Notion pages. Database operations benefit from structured data flow using `<function1.result>` for complex filter and sort parameters, allowing precise query execution based on workflow conditions.

The metadata outputs provide crucial operational context. Use `<notion1.metadata.url>` for sharing direct page links in notifications, `<notion1.metadata.totalResults>` for pagination logic in search operations, and `<notion1.metadata.properties>` for dynamic database schema analysis in complex workflows.

**Workflow Architecture Patterns**

Successful Notion workflows typically follow distinct architectural patterns optimized for specific use cases. The **Webhook â†’ Function â†’ Notion Create** pattern excels for automated documentation systems, where external events trigger structured content creation with predefined templates and formatting.

For content aggregation systems, implement **Schedule Trigger â†’ Notion Search â†’ Function â†’ Notion Write** workflows that regularly scan for specific content types, process them through business logic, and append summaries to master tracking pages. This pattern enables automated knowledge base maintenance and content curation.

Database-centric workflows benefit from **API Trigger â†’ Notion Query Database â†’ Agent â†’ Notion Create Page** patterns, where external data triggers database queries, AI processes the results for insights, and automatically generates summary pages or reports. This creates dynamic reporting systems that transform raw data into structured knowledge.

Interactive workflows leverage **Webhook â†’ Notion Read â†’ Agent â†’ Telegram** patterns for conversational interfaces that retrieve specific Notion content based on user queries and deliver formatted responses through messaging platforms.

**Message/Data Formatting Excellence**

Notion's content structure requires careful attention to formatting standards. When using `notion_write` operations, structure content using proper Markdown syntax with clear headers, bullet points, and code blocks. Notion interprets Markdown formatting natively, so consistent formatting patterns ensure professional-looking automated content.

For database operations, format filter parameters as valid JSON objects with proper property references. Use structured approaches like `{"property": "Status", "select": {"equals": "Active"}}` rather than ad-hoc string formatting. This prevents parsing errors and ensures reliable query execution.

Content creation workflows should validate input data before sending to Notion operations. Implement content length checks, special character handling, and template validation to maintain consistent page structure across automated content generation. Long-form content benefits from paragraph breaks and section headers that improve readability in Notion's interface.

When working with database queries, structure sort parameters as JSON arrays with clear property and direction specifications: `[{"property": "Created", "direction": "descending"}]`. This ensures predictable result ordering and enables reliable pagination handling in complex workflows.

**Debugging and Monitoring**

Comprehensive monitoring requires tracking both successful operations and failure conditions across all Notion block operations. Monitor `<notion1.content>` outputs for empty results that might indicate permission issues, deleted pages, or incorrect ID parameters. The content field provides descriptive error messages that pinpoint specific operational failures.

Implement metadata-based validation by checking `<notion1.metadata.totalResults>` for search and query operations. Zero results might indicate overly restrictive filters, incorrect database structures, or outdated query parameters. Use this data to implement fallback logic and error recovery patterns.

For database operations, monitor property schema changes by comparing `<notion1.metadata.properties>` across workflow executions. Notion database schema modifications can break existing filter and sort logic, so proactive monitoring prevents workflow failures when team members modify database structures.

Page creation operations require validation of `<notion1.metadata.pageId>` outputs to ensure successful creation before attempting subsequent operations. Failed page creation often results from insufficient permissions, invalid parent IDs, or workspace limitations that require immediate attention.

Performance monitoring should track operation execution times, particularly for large database queries and content-heavy read operations. Notion's API rate limits require careful workflow scheduling to avoid temporary blocks that could disrupt automated processes.

**Security Considerations**

Notion workspace security requires careful attention to access permissions and data handling practices. Ensure Notion API tokens used in Sim.ai workflows have minimal required permissions, limiting access to specific pages and databases rather than workspace-wide access. This reduces potential impact from credential compromise.

When handling sensitive content through Notion blocks, implement content filtering to prevent accidental exposure of confidential information. Use function blocks to sanitize AI-generated content before writing to shared Notion pages, removing personal identifiers, financial data, or proprietary information.

Database operations should validate filter parameters to prevent information disclosure through overly broad queries. Implement query validation that ensures users can only access data within their authorization scope, particularly in multi-tenant workflow environments.

Monitor page and database sharing settings when creating new content programmatically. Automated page creation should respect existing workspace sharing policies and avoid inadvertently creating publicly accessible content that should remain private.

**Performance Optimization**

Optimize Notion block performance through strategic caching and batch operation patterns. Implement read caching for frequently accessed pages using temporary storage blocks, reducing redundant API calls and improving workflow response times. Cache page content for workflow sessions rather than re-reading unchanged content multiple times.

Database query optimization requires careful filter and sort parameter design. Use specific property filters rather than broad search terms to reduce result set sizes and improve query performance. Implement pagination logic for large databases by processing results in manageable chunks using the `pageSize` parameter effectively.

Content creation workflows benefit from batching strategies where multiple updates accumulate before writing to Notion pages. Rather than appending individual items repeatedly, collect updates in temporary storage and perform bulk append operations to minimize API calls and improve overall throughput.

Search operations should use specific query terms and filter types to reduce processing overhead. Implement progressive search refinement where broad queries narrow down through subsequent operations rather than processing large result sets client-side.

For high-frequency workflows, implement operation queuing using schedule triggers to distribute Notion API calls across time intervals, preventing rate limit conflicts and ensuring consistent performance during peak usage periods.
